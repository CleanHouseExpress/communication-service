# Tenant Database Provisioning

## Landlord vs Tenant DB

The communication-service has a landlord database that stores service-level metadata:

- tenant replicas from Orchestra;
- tenant connection records;
- integration events;
- current communication tables until the tenant split is implemented.

Future tenant databases will store communication workload data per tenant/rede. This phase only prepares connection metadata and optional physical database creation.

## Database Naming

Tenant database names are generated by `TenantDatabaseNameGenerator`.

Pattern:

```txt
<COMMUNICATION_TENANT_DB_PREFIX><tenant-slug-or-orchestra-id>
```

Default:

```txt
communication_tenant_<slug>
```

Rules:

- lowercase only;
- letters, numbers and underscore only;
- dangerous characters are replaced;
- maximum length is 64 characters;
- the name is never accepted directly from the request payload.

Example:

```txt
slug: Rede Exemplo
database: communication_tenant_rede_exemplo
```

## Provisioning Flow

Manual endpoint:

`POST /api/internal/tenants/{orchestra_tenant_id}/provision-database`

Flow:

1. Validate service token.
2. Resolve active tenant by `orchestra_tenant_id`.
3. Generate safe database name.
4. If provisioning is disabled, create/update `communication_tenant_connections` as `skipped`.
5. If provisioning is enabled, create the MySQL database with a safely quoted generated name.
6. Create/update tenant connection metadata.
7. Return connection status without secrets.

Optional event integration:

- `COMMUNICATION_TENANT_DB_AUTO_PROVISION=true`
- when a `TenantCreated` event is processed, the service calls `ProvisionTenantDatabaseAction`.

## Configuration

```env
COMMUNICATION_TENANT_DB_PROVISIONING_ENABLED=false
COMMUNICATION_TENANT_DB_AUTO_PROVISION=false
COMMUNICATION_TENANT_DB_PREFIX=communication_tenant_
COMMUNICATION_TENANT_DB_HOST=
COMMUNICATION_TENANT_DB_PORT=3306
COMMUNICATION_TENANT_DB_USERNAME=
COMMUNICATION_TENANT_DB_PASSWORD=
COMMUNICATION_TENANT_DB_DRIVER=mysql
```

## Security

- Database name is generated from trusted tenant replica fields and sanitized.
- The raw slug/name is never interpolated into SQL.
- Physical database creation only runs when provisioning is explicitly enabled.
- Password is not returned in API responses.
- If stored, database password is encrypted via Laravel `Crypt`.
- Logs include tenant id, connection id, database name and status, but no password.

## Current Limitations

- No tenant migrations are executed.
- `migrated_at` remains null.
- Models still use the current landlord connection.
- No dynamic connection switching.
- No database deletion/drop flow.
- No queues or retries.
- MySQL database creation uses the app's current DB connection privileges.
